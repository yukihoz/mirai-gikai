# トピック分析・バックフィルの Google Cloud Run 実行基盤 設計書

ユーザー向けトピック分析（Step 3 の分析パイプライン）と意見再抽出バックフィル（Step 2）の**実行基盤を Vercel のフェーズ連鎖から Google Cloud Run Job に移行**するための設計書。分析・抽出の**ロジック本体は流用**し、実行の殻だけを差し替える。

- 対象: 分析（extract→merge→assign）＋バックフィル（再抽出）の**両方**。
- 実行形態: **Cloud Run Job**（バッチ・走り切り・タイムアウト実質なし・リトライ可）。
- 元の設計書 §5.3「将来の堅牢化（Cloud Run）」の具体化。フェーズ分割・状態機械はそのまま流用可、という想定どおり。

---

## 0. 背景と目的

現状の Step2/3 は Vercel のサーバーレス関数上で動き、5分制限を回避するために「`after()` で即レスポンス → 内部 fetch で次ステップを自己トリガ」という**フェーズ連鎖**で実行している。これは設計書 §5.3 が「一旦は追加インフラ無しで」と認めた簡易実装で、次の弱点がある（実装・レビューで確認済み）:

- **5分/呼び出しの制限**: 1チャンクを5分に収める必要があり、大規模議案では分割が必須。
- **ベストエフォートで耐障害性なし**: 内部 fetch（自己トリガ）が失敗するとチェーンが静かに停止し、自動再開しない。レスポンス喪失で誤って `failed` 表示 → 多重起動を招く懸念（codex 指摘 P2b）。
- **同時実行の制御が弱い**（codex 指摘 P2a）。
- 可観測性が低い（ログが複数の関数呼び出しに分散）。

**目的**: 分析・バックフィルを **1プロセスで完走**させ、5分制限・自己トリガ・分割を不要にし、リトライ・並列度・ログ・スケジュールを Cloud Run の機構に委ねて堅牢化する。

## 1. 全体アーキテクチャ

```
                       ┌─────────────────────────────────────────────┐
[Admin 実行ボタン] ──→ │ Admin API (Vercel)                          │
                       │  - requireAdmin / version 作成 (pending)     │
                       │  - Cloud Run Jobs Execute API を呼ぶ         │
                       └───────────────┬─────────────────────────────┘
                                       │ run.jobs.run (overrides: args)
[Cloud Scheduler] ──(OAuth)──────────→ ▼
                       ┌─────────────────────────────────────────────┐
                       │ Cloud Run Job: topic-analysis-worker         │
                       │  tsx src/main.js --mode=<analyze|backfill>   │
                       │   --bill-id / --version-id                   │
                       │  ・パイプライン全体を1プロセスで実行          │
                       │  ・LLM並列は in-process（mapWithConcurrency） │
                       │  ・5分制限なし / 自己トリガなし / リトライ可  │
                       └───────────────┬─────────────────────────────┘
                                       │ Supabase Secret Key
                                       ▼
                       Supabase (topic_analysis_version / topic /
                                 topic_opinion / interview_opinion)
                                       ▲
[Admin UI] ──(status ポーリング: DB の status/current_step)────────────┘
```

- **トリガ**: Admin から `run.jobs.run`（Cloud Run Admin API）でジョブ実行を起動。cron は Cloud Scheduler が同 API を叩く。
- **進捗**: worker が `topic_analysis_version.status / current_step / progress` を更新。Admin UI は**従来どおり DB をポーリング**（自己トリガが消えるだけで UI 側はほぼ無変更）。
- **完了/失敗**: ジョブの終了コード＋DB の status で判定。リトライは Cloud Run Job の `max-retries` に委譲。

## 2. コード再構成

### 2.1 新パッケージ `@mirai-gikai/topic-analysis-core`（packages/topic-analysis-core）

admin/src 配下にあるパイプライン本体を、worker と admin の両方から import できる共有パッケージへ抽出する（admin/src はコンテナから import 不可のため）。

**移設するもの（admin/src/features/* → packages/topic-analysis-core/src/*）:**

| 現在地 | 移設先 | 備考 |
| :-- | :-- | :-- |
| user-topic-analysis/shared/{types,constants,schemas} | core/shared | そのまま |
| user-topic-analysis/shared/utils/{parse-assignment-tsv, build-sorted-topics}(+test) | core/utils | 純粋関数・テストごと |
| user-topic-analysis/server/services/{extract,merge,assign,orchestrator,prompts} | core/services | analyze パイプライン |
| user-topic-analysis/server/repositories/user-topic-analysis-repository | core/repositories | createAdminClient ベース・@/ 依存なし |
| user-topic-analysis/server/utils/concurrency | core/utils | mapWithConcurrency/withRetry/chunk |
| interview-opinion-backfill/shared/{constants, utils/prepare-reextraction-messages}(+test) | core | backfill |
| interview-opinion-backfill/server/services/{reextract-report-opinions, run-opinion-backfill-chunk} | core/services | backfill |
| interview-opinion-backfill/server/repositories/interview-opinion-backfill-repository | core/repositories | backfill |

**core が新たに owns する小クエリ**（現在 admin @/ から借りているもの。createAdminClient で再実装）:
- `findInterviewConfigById` / `findInterviewSessionById` / `findInterviewMessagesBySessionId`（reextract が使用）
- `fetchBillWithContents`（reextract が使用。analyze 側の `fetchBillContext` と統合してもよい）

**依存**: `@mirai-gikai/supabase`, `@mirai-gikai/shared`, `ai`, `zod`。`server-only` は付けない（Node ワーカー＝サーバー実行が前提のため）。`@/lib/env` 依存（trigger-step / trigger-backfill-run）は**フェーズ連鎖の撤去で不要**になり移設しない。

### 2.2 worker アプリ（worker/）

Cloud Run Job 用の最小 Node アプリ。

```
worker/
├── package.json        # deps: @mirai-gikai/topic-analysis-core, tsx/esbuild など
├── Dockerfile
├── src/main.ts         # 引数を解釈し core の orchestrator を呼ぶ
```

`src/main.ts`（擬似コード）:
```ts
import { runAnalysis } from "@mirai-gikai/topic-analysis-core/analyze";
import { runBackfill } from "@mirai-gikai/topic-analysis-core/backfill";

const mode = arg("--mode");            // "analyze" | "backfill" | "cron"
const billId = arg("--bill-id");
const versionId = arg("--version-id");

async function main() {
  if (mode === "analyze") await runAnalysis({ billId, versionId });          // extract→merge→assign→finalize を直列実行
  else if (mode === "backfill") await runBackfill();                          // 残りを全件処理（チャンクループ）
  else if (mode === "cron") await dispatchEligibleBills();                    // §6トリガ条件で対象billを選び順次 analyze
  else throw new Error(`unknown mode: ${mode}`);
}
main().then(() => process.exit(0)).catch((e) => { console.error(e); process.exit(1); });
```

- `runAnalysis` は現在の executeExtract→executeMerge→executeAssign を**1関数で直列実行**（フェーズ連鎖を排除）。version の status/current_step を更新しつつ進める。失敗は version を failed にし非0終了。
- `runBackfill` は現在の run-opinion-backfill-chunk を**残りが無くなるまでループ**（自己トリガ撤去）。

### 2.3 admin の簡素化

- **削除**: `/api/user-topic-analysis/{dispatch,steps/*}`、`/api/interview-opinion-backfill/{dispatch,run}`、`trigger-step` / `trigger-backfill-run` / `handle-step-route`、middleware の内部ルート bypass（不要に）。
- **新設**: `/api/user-topic-analysis/run`（requireAdmin → version 作成 → Jobs Execute）と `/api/interview-opinion-backfill/run`（Jobs Execute）。status ルートは存続（DB読み取り）。
- **UI**: 実行ボタン・進捗ポーリング・結果ビューは存続（トリガ先が Cloud Run になるだけ）。
- admin はパイプライン本体を持たず、必要な読み取り（結果ビュー）は core のリポジトリを import。

## 3. トリガ設計（Admin → Cloud Run Jobs）

Admin（Vercel/Node）から **Cloud Run Admin API** の `projects.locations.jobs.run` を呼んでジョブ実行を起動し、`overrides.containerOverrides.args` で `--mode/--bill-id/--version-id` を渡す。

```
POST https://run.googleapis.com/v2/projects/{PROJECT}/locations/{REGION}/jobs/{JOB}:run
Authorization: Bearer <SA access token>
{ "overrides": { "containerOverrides": [{ "args": ["--mode=analyze","--bill-id=<uuid>","--version-id=<uuid>"] }] } }
```

- 認証: GCP サービスアカウント（`roles/run.developer` 相当）で OAuth アクセストークンを取得して Bearer 送信。`google-auth-library` を admin に追加。
- cron: **Cloud Scheduler**（OAuth）が同 API を `--mode=cron` で叩く（§6 のトリガ条件は worker 側で判定）。

## 4. GCP リソース

| リソース | 用途 |
| :-- | :-- |
| Artifact Registry（docker repo） | worker イメージ格納 |
| Cloud Run Job `topic-analysis-worker` | パイプライン実行。`--max-retries`、`--task-timeout`（例 60m）、メモリ/CPU 設定 |
| Secret Manager | `SUPABASE_URL` / `SUPABASE_SECRET_KEY` / `AI_GATEWAY_API_KEY`（Job にマウント） |
| Service Account（worker 実行用） | Secret 参照権限 |
| Service Account（admin 起動用） | `run.jobs.run` 権限。鍵を Vercel env に格納（or Workload Identity Federation） |
| Cloud Scheduler | 日次 cron → Job 実行（`--mode=cron`） |

**gcloud（概略・あなたが実行）:**
```sh
gcloud artifacts repositories create topic-analysis --repository-format=docker --location=$REGION
# build & push（CIでも可）
gcloud run jobs create topic-analysis-worker \
  --image=$REGION-docker.pkg.dev/$PROJECT/topic-analysis/topic-analysis-worker:latest \
  --region=$REGION --task-timeout=3600 --max-retries=1 --memory=1Gi \
  --set-secrets=SUPABASE_SECRET_KEY=...:latest,AI_GATEWAY_API_KEY=...:latest \
  --set-env-vars=SUPABASE_URL=...
# 起動用SAに権限
gcloud run jobs add-iam-policy-binding topic-analysis-worker --member=serviceAccount:$ADMIN_SA --role=roles/run.developer --region=$REGION
```

## 5. CI（GitHub Actions）

既存 `deploy.yml` とは別に worker 用ワークフローを追加:
1. worker イメージを build → Artifact Registry に push（`google-github-actions/auth` + WIF が望ましい）。
2. `gcloud run jobs update topic-analysis-worker --image=...:$SHA` でイメージ更新。

**実行方式（確定）**: ビルド工程（dist 生成）は持たず、`tsx` で `.ts` を直接実行する。
core パッケージの `exports` も `.ts` を指したままで問題ない（worker は `node --conditions=react-server` 下で `tsx` 起動し、shared の `import "server-only"` を空 stub に解決させる）。Dockerfile は `pnpm install --filter @mirai-gikai/topic-analysis-worker...` で core を含む依存だけを入れ、`pnpm exec tsx src/main.ts` を ENTRYPOINT にする。

## 6. ローカル開発・テスト

- worker は `node` で直接実行可能: `dotenv -e .env -- pnpm --filter worker exec tsx src/main.ts --mode=analyze --bill-id=...`。ローカル Supabase に対して end-to-end 確認できる（5分制限もフェーズ連鎖も無いので素直）。
- core の純粋関数テスト（parse-assignment-tsv / build-sorted-topics / prepare-reextraction-messages / enrich など）と統合テスト（§8フィルタ・version連番・topic_opinion PK・dual-write）は **packages 側に移して存続**。
- admin の API は「Jobs Execute を呼ぶ薄い層」になるため、Jobs クライアントを DI でモックして単体テスト可能。

## 7. 移行・ロールアウト計画

1. `@mirai-gikai/topic-analysis-core` を作成し、パイプライン＋リポジトリ＋テストを admin から移設（admin は core を re-import して一旦グリーンを維持）。
2. worker アプリ＋Dockerfile＋エントリポイント作成。ローカルで analyze/backfill を end-to-end 検証。
3. admin のトリガを Vercel フェーズ連鎖 → Jobs Execute に差し替え。フェーズ連鎖の route/util/middleware bypass を削除。
4. GCP リソース作成（あなた）＋ CI でイメージ deploy。本番ジョブで 1 議案を analyze 実行して検証。
5. cron（Scheduler）を有効化（パイロット議案から）。

DB スキーマ（version/topic/topic_opinion/opinions_reextracted_at）は**無変更**。

## 9. 決定事項・残論点

**確定（本書）:**
- 実行形態 = Cloud Run **Job**。
- 対象 = 分析（Step3）＋バックフィル（Step2）両方。
- パイプライン本体は `@mirai-gikai/topic-analysis-core` に抽出し worker/admin で共有。
- Vercel フェーズ連鎖（dispatch/steps/自己トリガ/middleware bypass）は**撤去**。
- DB スキーマ無変更。

**確定（ユーザー確認済み 2026-06-09）:**
- **admin → Jobs 認証 = サービスアカウント鍵を Vercel env に格納**（`google-auth-library` でトークン取得）。WIF は将来の堅牢化候補。
- **cron は今回スコープ外**（まず手動トリガのみ。Cloud Scheduler／§6 トリガ条件は次フェーズ。worker は `--mode=analyze|backfill` のみ実装し、`cron` は後追い）。
- **worker 置き場所 = `worker/`（トップレベル app）**、core パッケージ名 = `@mirai-gikai/topic-analysis-core`（暫定）。
- 既存 #826/#827 の実行レイヤーは本ブランチ（`topic-analysis-cloud-run`・step3 にスタック）で差し替える。

**確定（GCP）:**
- プロジェクト ID = `<your-gcp-project-id>`（OSS のため実値はコミットしない）、リージョン = `asia-northeast1`（東京）。
- コードは env（`GCP_PROJECT_ID=<your-gcp-project-id>` / `GCP_REGION=asia-northeast1` / `GCP_TOPIC_ANALYSIS_JOB=topic-analysis-worker` / `GCP_SA_KEY=<JSON>`）で参照し、実プロビジョニング（プロジェクト作成・SA・Secret・初回 deploy）はあなたが手順書に沿って実施。

## 10. 想定される効果

- 5分制限・チャンク分割・自己トリガ・middleware bypass が**すべて不要**に。
- 大規模議案（数千意見）でも 1 実行で完走。LLM 並列は in-process で維持。
- リトライ・タイムアウト・ログ・スケジュールを Cloud Run/Scheduler の標準機構に委譲。
- codex 指摘 P2a（同時実行）/P2b（自己トリガ失敗の誤 failed）が**構造的に解消**。
