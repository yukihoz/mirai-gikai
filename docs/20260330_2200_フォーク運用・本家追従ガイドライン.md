# フォーク運用・本家追従ガイドライン

## 1. 目的

「みらい議会@中央区」プロジェクトは、大元のOSSである `team-mirai/mirai-gikai` をベースとしています。
本家リポジトリの改善（新機能、バグ修正、セキュリティアップデート）を継続的に取り込みつつ、中央区独自のカスタマイズを維持・拡張するための運用方針を定めます。

## 2. Git（上流追従）の基本設定

ローカルの開発環境では、常に自身のフォーク元（`origin`）と大元のリポジトリ（`upstream`）の両方をトラックします。

### 2.1 リモートリポジトリの追加
開発を開始する前に、以下のコマンドで大元リポジトリを `upstream` として追加してください。
```bash
git remote add upstream https://github.com/team-mirai/mirai-gikai.git
```

### 2.2 本家の更新の取り込みフロー
定期的に（または本家で重要な更新があった際に）、以下の手順で本家の `main` ブランチの変更を取り込みます。
```bash
git fetch upstream
git merge upstream/main
# コンフリクトが発生した場合は解消し、コミットします
```

## 3. 中央区独自改修の設計ルール

アップデート時のコンフリクト（競合）を最小限に抑え、保守性を高く保つため、以下のルールに従って機能追加・改修を行います。

### 3.1 既存コードの変更は最小限にする
- **環境変数の活用**: ロゴ画像、サイト名、ベースカラーなどの設定値は、可能な限り環境変数 (`.env`) や外部設定ファイルに切り出し、大元のソースコード自体（ハードコードされたテキスト）を直接書き換えるのを避けます。
- **コンポーネントの分離**: 独自の複雑なUIや機能が必要になった場合は、既存のファイルを大きく改修するのではなく、`xxx-chuo.tsx` のような新規コンポーネントファイルを作成し、呼び出し元だけを差し替えるアプローチを取ります。

### 3.2 データベース（Supabase）の変更ルール
- **マイグレーションの追記**: 新しいテーブルやカラム（例: `meeting_body`）を追加する場合は、絶対に**過去のマイグレーションファイルを直接書き換えない**でください。
- 必ず以下のコマンドで新しいマイグレーションファイルを作成し、差分だけを定義して適用します。
  ```bash
  npx supabase migration new add_feature_xxx
  ```

### 3.3 汎用的な機能は本家（Upstream）に還元する
- 今回行った「衆議院・参議院（originating_house）」を「会議体（meeting_body）」へ抽象化するような改修は、他の地方自治体版でも使える**汎用的な改善**です。
- このような「本家にもメリットがある改修」や「バグ修正」を行った場合は、独自のブランチに抱え込まず、積極的に `team-mirai/mirai-gikai` に対して Pull Request（PR）を送ることを目指します。
- 本家に取り込まれることで、自分たちが将来保守しなければならない「独自差分」が減り、運用の手間が大幅に削減されます。

## 4. 運用・タスクの進め方
今後、新しい機能の要望や改修案が出た場合は、まず本ドキュメントの「3. 設計ルール」に照らし合わせ、**「いかに本家との差分を綺麗に保ちながら実現するか」**というアプローチで実装方針を検討・決定するものとします。
