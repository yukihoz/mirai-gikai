# 地方議会・自治体版カスタマイズガイド

本ドキュメントは、「みらい議会」プラットフォームを特定の自治体（区議会・市議会等）向けにカスタマイズし、かつ本家リポジトリ（upstream）とのマージ互換性を維持するための設計・設定ガイドです。

---

## 1. サイト基本情報の定義（環境変数）

自治体名やサイトの名称はエンジンのコアコードを直接書き換えるのではなく、環境変数（`.env`）を通じて管理します。

追加された主な環境変数：
- `NEXT_PUBLIC_SITE_TITLE`: サイトの正式名称（例：「みらい議会@中央区（仮）」）
- `NEXT_PUBLIC_SITE_SHORT_NAME`: サイトの略称（例：「みらい議会」）
- `NEXT_PUBLIC_ASSEMBLY_NAME`: 議会名称（例：「区議会」）
- `NEXT_PUBLIC_TEAM_NAME`: 運営チーム名（例：「チームみらい」）

> [!NOTE]
> これにより、ソースコード側は `env.assemblyName` のように参照するだけでよくなり、他の自治体へ展開する際も `.env` の書き換えのみで対応可能になります。

---

## 2. AIシステムプロンプトの汎用化戦略

特定の議員名、議席数、地域特有の課題などをAIプロンプト（`shared-sections.ts` 等）に直接書き込むのは避けてください。

### 知識の分離（Knowledge Source）
AIに特定の自治体の情報を教えたい場合は、以下の優先順位で情報を与えます：

1.  **ナレッジソース（RAG）**: 議員の情報、過去の会議録、自治体独自の制度などは、Supabaseのドキュメント（Knowledge Table）等に投入し、AIが検索して回答できるようにします。
2.  **プレースホルダー**: どうしてもプロンプトに含める必要がある基本情報は、環境変数（`${env.teamName}` 等）を埋め込む形式に留めます。

> [!TIP]
> 「2025年プラン」のような時間経過とともに古くなる情報も、ソースコードから分離して管理することで、システムメンテナンスの負荷を軽減できます。

## 3. ステータス名称の地方議会向け調整

国会（二院制）ベースの名称を地方議会（一院制・委員会重視）のプロセスへ適合させています。

| 内部コード | 地方議会向けラベル | 備考 |
| :--- | :--- | :--- |
| `introduced` | **議案提出済み** | 旧：提出済み |
| `in_originating_house` | **{会議体}付託** | 旧：{会議体}審議中 |
| `enacted` | **議案可決** | 旧：成立 |
| `rejected` | **議案否決** | 旧：否決 |
| `reported` | **報告事項** | 旧：報告済み |

### 実装上の注意
- 表示ロジック：`web/src/features/bills/shared/types/index.ts` の `getBillStatusLabel`
- カード用ラベル：`web/src/features/bills/shared/utils/bill-status.ts` の `getCardStatusLabel`
- 入力フォーム：`admin/src/features/bills-edit/client/components/bill-form-fields.tsx` の `BILL_STATUS_OPTIONS`

## 4. 本家リポジトリ（Upstream）との付き合い方

将来的な機能追加（AIインタビュー機能の統合等）をスムーズに取り込むため、以下の設計指針を遵守します：

- **既存ファイルの破壊的変更を避ける**: 可能な限り既存の型定義や関数シグネチャを維持し、表示文字列のみを変更します。
- **ロジックの追加は新規ファイルで**: コンポーネントの構造を大きく変える必要がある場合は、別のコンポーネントとして作成し、それを利用する形を検討してください。
- **環境変数のデフォルト設定**: 新しい環境変数を追加した際は、必ず `.env.example` にも追加し、デフォルト値を設定してください。
