# 増分トピック分析（インクリメンタル抽出・差分追加）設計

## 1. 背景・目的
- 現状、ユーザー向けトピック分析の2回目以降は **全意見を再抽出・再分類（フル）** している（Phase1 extract→Phase2 merge→Phase3 assign を全意見対象で実行し、新 version を作る）。
- インタビューは日々増えるため、毎回ゼロから抽出するのはコストが高い。
- そこで **「新規意見だけを抽出し、既存トピックへ差分追加する」増分モード** を追加する（フルモードは残す）。
- 想定運用: 全議案ページから**日次で増分実行**（低コスト）、フル再分析は手動 per-bill で残す。

## 2. 現状アーキテクチャ（要約）
- `topic_analysis_version`（議案ごと・version連番）/ `topic`（version_id単位）/ `topic_opinion`（version_id単位・opinion↔topic）。
- パイプライン（`packages/topic-analysis-core`）: `extractTopics`(全意見) → `mergeTopics` → `assignOpinions`(全意見) → `saveTopicsAndAssignments` → `finalizeVersion` → `publishVersion`。
- worker: `--mode=analyze --bill-id=<uuid> --version-id=<uuid>`（単一議案）。version 行は admin の dispatch が `createVersion` で先に作る。
- 対象意見: `fetchTargetOpinions(billId)` = interview_opinion を `interview_report(is_public_by_admin & is_public_by_user & moderation_status='ok')` でフィルタ。

## 3. データモデル変更
### 3.1 抽出ウォーターマーク列の追加
```sql
ALTER TABLE interview_opinion
  ADD COLUMN topic_extracted_at TIMESTAMPTZ;
COMMENT ON COLUMN interview_opinion.topic_extracted_at IS
  'トピック抽出(Phase1)の対象に含めた時刻。NULL=未抽出(増分対象の新規意見)';
```
- **NULL = 「まだ一度もトピック抽出にかけられていない新規意見」**。
- 増分の extract ステップで対象に含めた意見に `now()` を記録する。
- 型再生成（`packages/supabase/types`）をセットでコミット。

### 3.2 backfill（意見再抽出）との相互作用
- `syncInterviewOpinions` は `(interview_report_id, opinion_index)` で upsert する。`buildInterviewOpinionRows` は `topic_extracted_at` を**出力カラムに含めない**ため:
  - 新規意見（新しい report/index）→ INSERT で `topic_extracted_at = NULL`（＝新規扱い・正）。
  - 既存意見の再抽出 → 既存行の `topic_extracted_at` は **upsert で更新されず保持**（＝再抽出しても「抽出済み」のまま・正）。
- 注意点（論点）: 再抽出で意見内容が変わっても `topic_extracted_at` は保持される。＝「再抽出した意見」は増分の新規対象にはならない（フル実行でのみ反映）。これは意図通り（増分は“新しく増えた意見”の取り込みが目的）。

## 4. 増分パイプライン
前回の最新 completed version を **V（例 v1）**、新しく作る version を **V'（例 v2）** とする。

### Step 0: V' を作成し、V を引き継ぐ（carry-forward）
- `createVersion(billId, trigger, model, promptVersion)` で V' を作成。
- **V の `topic` を V' に複製**（version_id=V' で再 insert、新トピックIDを採番。title/description/sort_order を保持）。
- **V の `topic_opinion` を V' に複製**（旧topicID→新topicID のマッピングで付け替え）。
- → V' は V の完全コピーから始まる（既存トピック・既存割当を引き継ぐ）。

### Step 1: 新規意見のみ抽出（extract-new）
- このbillの対象意見のうち **`topic_extracted_at IS NULL`** のものだけを取得。
- 0件ならトピック追加・割当もスキップして finalize（V' は実質 V のコピー）。コストを使わない。
- Phase1 と同じ `extractTopics` を新規意見に対して実行 → 新トピック候補。
- 抽出対象に含めた意見の `topic_extracted_at = now()` を更新。

### Step 2: 既存トピックとの突き合わせ（diff / accept）
- 新候補を **V' の既存トピック群と LLM で突き合わせ**、各候補について採否判定:
  - 採用: 既存のどれとも主張の方向・対象分野・論点が**明確に異なり**、かつ fine 粒度方針に合う。
  - 不採用（破棄）: 既存トピックに該当する／粒度が不適切／**判定が曖昧なものは不採用に倒す**（トピック乱立を防ぐ）。
- 採用された候補のみ V' に**新トピックとして追加**（sort_order 採番は finalize 時に件数で再ソート）。

### Step 3: 未割当意見の割当（assign）
- **V' で未割当の意見** = 対象意見全体 − V' に既に割当済み（carry-forward 分）= 新規意見＋前回未割当意見。
- これらを **V' の全トピック（既存＋新規）** に対して `assignOpinions` で割当。
- 既存トピックにも割り当て得る（新規意見が既存トピックに入るのは正常）。
- topic_opinion に追加（既存割当は不変）。

### finalize / publish
- 件数降順で sort_order 再計算 → `saveTopicsAndAssignments` 相当 → `finalizeVersion` → `publishVersion`（既定どおり自動公開）。

### 既存トピックの扱い（凍結）
- 既存トピックの **title / description（3行要約）/ 既存の割当は再生成・変更しない**（コスト削減の主眼）。
- 既存トピックに新規意見が増えても説明は更新しない（**一旦フリーズ**。将来オプションで再生成を検討）。

## 5. worker モード / 起動パラメータ
- `--mode=analyze` に `--strategy=full|incremental` を追加（**既定 `full`＝後方互換**）。
  - `full`: 現状どおり全意見で再分析。
  - `incremental`: 本設計の Step0〜3。
- 全議案: `--mode=analyze-all --strategy=incremental`（対象議案を列挙し、新規意見がある議案のみ順次・増分実行。1 Cloud Run 実行で全議案ループ）。
- `one_active_version_per_bill` で多重起動を防止。

## 6. admin UI / API
- **per-bill**: 既存の実行ボタンに「フル / 差分」の選択を追加（or 差分ボタン併設）。dispatch に `strategy` を渡す。
- **全議案ページ（新規）**: `/user-topic-analysis`（全議案）に「全議案トピック分析（差分）を実行」ボタン。`POST /api/user-topic-analysis/dispatch-all` → `executeTopicAnalysisJob(["--mode=analyze-all","--strategy=incremental"])`。バックフィル runner UI に倣う（進捗は対象数/完了数の粗い表示）。

## 7. ガード・コスト
- 新規意見0件の議案は version を作らずスキップ。
- 全議案×増分でも、各議案は新規分のみ抽出＝低コスト。
- 全議案ループは1ジョブ内順次（並列起動しない）。

## 8. 検証方法（ローカル）
1. 防災庁等で **フル**実行 → v1（全トピック・全意見 `topic_extracted_at` セット）。
2. 新規 report/意見を数件追加（or backfill で新規 report 取り込み）→ それらは `topic_extracted_at IS NULL`。
3. **差分**実行 → v2 を確認:
   - v2 のトピック = v1 トピック（同 title/description）＋（あれば）新トピック。
   - 既存トピックの内容が不変。
   - 新規意見が（新規 or 既存）トピックに割り当てられている。
   - 新規意見の `topic_extracted_at` が記録されている。
4. 新規意見0件で差分実行 → 新トピック0・コスト最小で完了。
- 純粋関数（候補×既存の突き合わせ整形、未割当抽出 等）には unit テスト。DB function は追加しない（アプリ層）。

## 9. 未解決・将来課題
- 既存トピックに新規意見が増えた際の **説明（3行）の再生成**（現状フリーズ→将来オプション）。
- 既存トピックの統廃合（増分では行わない。肥大化したら手動フルで再構成）。
- 再抽出（backfill）で内容が変わった意見の増分への反映（現状は対象外＝フルで反映）。

## 10. 実装順序（予定）
1. マイグレーション（`topic_extracted_at`）＋型再生成。
2. core: 増分パイプライン（carry-forward / extract-new / diff-accept / assign-unassigned）＋純粋関数＋テスト。
3. worker: `--strategy` と `--mode=analyze-all`。
4. admin: dispatch（strategy）/ dispatch-all / 全議案ページ UI。
5. ローカル検証（防災庁・学校教育法）。
