# トピック分析 Cloud Run Job プロビジョニング手順

トピック分析・意見再抽出バックフィルの実行基盤（Cloud Run Job）を新規に構築する手順。
設計の背景は [20260609_1200_トピック分析CloudRun実行基盤設計.md](20260609_1200_トピック分析CloudRun実行基盤設計.md) を参照。

- GCP プロジェクト: `<your-gcp-project-id>`（実値は OSS のためここには記載しない）
- リージョン: `asia-northeast1`（東京）
- Artifact Registry リポジトリ: `topic-analysis`
- Cloud Run Job 名: `topic-analysis-worker`

以下はすべて `gcloud` で実行する。`PROJECT_ID` / `REGION` は環境に合わせて読み替える。

```bash
export PROJECT_ID=<your-gcp-project-id>
export REGION=asia-northeast1
export AR_REPO=topic-analysis
export JOB=topic-analysis-worker
gcloud config set project "$PROJECT_ID"
```

## 1. API 有効化

```bash
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com
```

## 2. Artifact Registry リポジトリ作成

```bash
gcloud artifacts repositories create "$AR_REPO" \
  --repository-format=docker \
  --location="$REGION" \
  --description="Topic analysis worker images"
```

## 3. Secret Manager にシークレット登録

worker は次の3つのenvを必要とする。値は本番 Supabase / AI Gateway のもの。

```bash
printf '%s' "https://<your-project>.supabase.co" | \
  gcloud secrets create SUPABASE_URL --data-file=-
printf '%s' "<supabase-secret-key>" | \
  gcloud secrets create SUPABASE_SECRET_KEY --data-file=-
printf '%s' "<ai-gateway-api-key>" | \
  gcloud secrets create AI_GATEWAY_API_KEY --data-file=-
```

> 値を更新する場合は `gcloud secrets versions add <NAME> --data-file=-`。

## 4. 実行用サービスアカウント（runtime SA）

Job 実行時にシークレットを読むための SA。

```bash
gcloud iam service-accounts create topic-analysis-runtime \
  --display-name="Topic Analysis Worker Runtime"

export RUNTIME_SA="topic-analysis-runtime@${PROJECT_ID}.iam.gserviceaccount.com"

# 各シークレットへの読み取り権限
for S in SUPABASE_URL SUPABASE_SECRET_KEY AI_GATEWAY_API_KEY; do
  gcloud secrets add-iam-policy-binding "$S" \
    --member="serviceAccount:${RUNTIME_SA}" \
    --role="roles/secretmanager.secretAccessor"
done
```

## 5. Cloud Run Job 作成（初回）

CI（`deploy_worker.yml`）はイメージ差し替え（`jobs update --image`）のみを行うため、
**Job 本体は初回だけ手動で作成する**。まず初回イメージをビルド & push する。

```bash
# リポジトリルートで実行（worker/Dockerfile はルートをビルドコンテキストにする）
gcloud auth configure-docker "${REGION}-docker.pkg.dev" --quiet
IMAGE="${REGION}-docker.pkg.dev/${PROJECT_ID}/${AR_REPO}/topic-analysis-worker:bootstrap"
docker build -f worker/Dockerfile -t "$IMAGE" .
docker push "$IMAGE"
```

Job を作成する。バッチ処理なので並列タスクは1・リトライ少なめ・タイムアウト長めにする。

```bash
gcloud run jobs create "$JOB" \
  --image "$IMAGE" \
  --region "$REGION" \
  --service-account "$RUNTIME_SA" \
  --set-secrets "SUPABASE_URL=SUPABASE_URL:latest,SUPABASE_SECRET_KEY=SUPABASE_SECRET_KEY:latest,AI_GATEWAY_API_KEY=AI_GATEWAY_API_KEY:latest" \
  --max-retries 1 \
  --task-timeout 3600 \
  --tasks 1 \
  --parallelism 1 \
  --cpu 1 \
  --memory 1Gi
```

> 引数（`--mode` 等）は固定せず、実行時に admin / CI が `--args` で渡す。

### 動作確認（手動実行）

```bash
# バックフィル
gcloud run jobs execute "$JOB" --region "$REGION" \
  --args="--mode=backfill"

# 分析（version は admin が作成するが、手動確認なら既存の version-id/bill-id を渡す）
gcloud run jobs execute "$JOB" --region "$REGION" \
  --args="--mode=analyze,--bill-id=<uuid>,--version-id=<uuid>"

# ログ
gcloud run jobs executions list --job "$JOB" --region "$REGION"
```

## 6. admin（Vercel）から Job を起動するための SA

admin の API ルートが `run.jobs.run` を呼ぶための SA とキー。

```bash
gcloud iam service-accounts create topic-analysis-invoker \
  --display-name="Topic Analysis Job Invoker (admin)"

export INVOKER_SA="topic-analysis-invoker@${PROJECT_ID}.iam.gserviceaccount.com"

# Job を起動する権限
gcloud run jobs add-iam-policy-binding "$JOB" \
  --region "$REGION" \
  --member="serviceAccount:${INVOKER_SA}" \
  --role="roles/run.invoker"

# 実行時に runtime SA として振る舞うため actAs を許可
gcloud iam service-accounts add-iam-policy-binding "$RUNTIME_SA" \
  --member="serviceAccount:${INVOKER_SA}" \
  --role="roles/iam.serviceAccountUser"

# キー発行（この JSON を Vercel の GCP_SA_KEY に丸ごと格納する）
gcloud iam service-accounts keys create invoker-key.json \
  --iam-account="$INVOKER_SA"
```

### Vercel（admin）環境変数

| 変数 | 値 |
| --- | --- |
| `GCP_PROJECT_ID` | `<your-gcp-project-id>` |
| `GCP_REGION` | `asia-northeast1` |
| `GCP_TOPIC_ANALYSIS_JOB` | `topic-analysis-worker` |
| `GCP_SA_KEY` | `invoker-key.json` の中身（JSON 文字列） |

## 7. CI（GitHub Actions）用 SA

`deploy_worker.yml` がイメージを push し Job を更新するための SA とキー。

```bash
gcloud iam service-accounts create topic-analysis-deployer \
  --display-name="Topic Analysis Worker Deployer (CI)"

export DEPLOYER_SA="topic-analysis-deployer@${PROJECT_ID}.iam.gserviceaccount.com"

# イメージ push
gcloud artifacts repositories add-iam-policy-binding "$AR_REPO" \
  --location="$REGION" \
  --member="serviceAccount:${DEPLOYER_SA}" \
  --role="roles/artifactregistry.writer"

# Job 更新（run.jobs.update を含む）
gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:${DEPLOYER_SA}" \
  --role="roles/run.developer"

# Job が runtime SA を使うため actAs を許可
gcloud iam service-accounts add-iam-policy-binding "$RUNTIME_SA" \
  --member="serviceAccount:${DEPLOYER_SA}" \
  --role="roles/iam.serviceAccountUser"

# キー発行（GitHub Secrets の GCP_SA_KEY に格納）
gcloud iam service-accounts keys create deployer-key.json \
  --iam-account="$DEPLOYER_SA"
```

### GitHub（リポジトリ / Environment）設定

- Secret: `GCP_SA_KEY` = `deployer-key.json` の中身
- Variables（任意・デフォルトはワークフローに埋め込み済み）:
  `GCP_PROJECT_ID` / `GCP_REGION` / `GCP_AR_REPO` / `GCP_TOPIC_ANALYSIS_JOB`

`develop` / `main` への push で `worker/` 等が変わると、CI がイメージを差し替える。

## 8. 後片付け

発行した鍵ファイル（`invoker-key.json` / `deployer-key.json`）は各シークレットへ登録後に
ローカルから削除する。

```bash
rm -f invoker-key.json deployer-key.json
```
